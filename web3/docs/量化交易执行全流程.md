# 量化交易执行全流程

本文档记录当前项目从监控规则创建到订单记录生成的完整执行链路。当前阶段以半自动确认和模拟下单为主，真实交易能力仍以后续版本接入为准。

## 1. 新建监控规则

页面：`/rules` 监控规则
入口：新增规则抽屉
数据表：`monitor_rules`

用户在规则表单中配置交易所、交易对、触发方向、目标价、下单方向、订单类型、交易金额或数量、滑点、冷却时间、最大触发次数、是否模拟下单、是否启用。

## 2. 后台策略扫描规则

服务：`StrategyService`
核心输入：`monitor_rules` 中已启用的规则
核心输出：命中后进入信号层

后台每 1 秒进入一次扫描循环，每条规则再按自己的 `checkIntervalMs` 控制实际检测频率。

### 2.1 获取最新行情

系统根据规则的 `exchange` 和 `symbol` 获取最新市场价 `marketPrice`。

### 2.2 检查最大触发次数

系统检查 `triggeredCount` 是否已经达到 `maxTriggerCount`。

如果已达到上限：

```text
runtimeStatus = limit_reached
本轮结束
不会继续产生新信号
```

### 2.3 检查冷却时间

系统检查距离上次触发 `lastTriggeredAt` 是否已经超过 `cooldownMs`。

如果仍在冷却期内，本轮跳过。

### 2.4 比较市场价和目标价

根据 `operator` 比较市场价和目标价：

```text
gte：marketPrice >= targetPrice
lte：marketPrice <= targetPrice
```

如果条件未命中，本轮结束。
如果条件命中，进入信号层。

## 3. 生成交易信号

页面：`/signals` 交易信号
数据表：`trading_signals`
服务：`SignalService`

交易信号记录的是系统为什么认为当前可以交易。它是策略命中后的第一层业务记录。

信号生成时会记录市场价、目标价、触发方向、下单方向、订单类型、交易金额或数量、信号原因等信息。

当前实现中，信号生成后会立刻进入风控检查，并尝试转换成触发事件，所以 `pending` 通常停留时间很短。

## 4. 风控检查

页面：`/signals` 交易信号页右侧风控检查抽屉
数据表：`risk_checks`
服务：`RiskService`

风控检查负责判断信号是否允许继续进入人工确认阶段。

当前主要检查项包括：

| 检查项 | 含义 |
| --- | --- |
| 单笔最大金额 | 当前交易金额不能超过风控配置的单笔上限 |
| 行情新鲜度 | 行情事件时间不能超过允许延迟 |
| 每日最大次数 | 当日通过风控的次数不能超过配置上限 |
| 每日最大金额 | 当日通过风控的累计金额不能超过配置上限 |
| 交易模式 | 真实交易需要后端配置允许，当前默认以模拟为主 |

如果风控拒绝：

```text
trading_signals.status = rejected
流程停止
不会进入触发确认页
```

如果风控通过，继续生成触发确认。

## 5. 生成触发确认

页面：`/triggers` 触发确认
数据表：`trigger_events`

触发确认是人工执行闸门。系统只生成待确认事件，不会直接下单。

生成触发事件后，规则会更新：

```text
lastTriggeredAt = 当前触发时间
triggeredCount = triggeredCount + 1
runtimeStatus = running 或 limit_reached
```

如果新的 `triggeredCount` 已达到 `maxTriggerCount`，规则状态会变成 `limit_reached`。

## 6. 下单前预览

页面：`/triggers` 点击确认时弹窗
接口：`POST /api/orders/preview`

下单前预览由后端生成，前端只负责展示。真正计算和校验以后端结果为准。

如果交易规则或风控预览未通过，前端不允许继续确认下单。

## 7. 人工确认后生成订单记录

页面：`/triggers` 触发确认页右侧订单记录抽屉
数据表：`order_records`
接口：`POST /api/orders/confirm`

用户在触发确认页完成确认后，后端执行下单流程，并写入订单记录。

当前阶段主要是模拟下单：

```text
生成本地 exchangeOrderId
订单状态写为 submitted
记录 rawMessage
```
## 8. 审计日志贯穿全流程

页面：`/audit-logs` 审计日志
数据表：`audit_logs`

审计日志用于记录关键业务动作和异常，方便回溯完整执行链路。

## 整体链路

```text
/rules 新建规则
-> 后台策略扫描
-> /signals 生成交易信号
-> /signals 内查看风控检查
-> /triggers 生成待确认触发
-> /triggers 下单前预览
-> /triggers 内查看订单记录
-> /audit-logs 记录全流程审计
```

## 项目特点与交易所内置功能区别

本项目的定位是本地量化交易执行体。它管理的是一笔交易从规则、信号、风控、确认、预览到订单记录的完整执行过程。

欧易和币安也提供限价单、条件单、止盈止损、交易机器人、API 下单和订单历史等能力，但这些能力大多运行在交易所平台内，偏向交易所内置工具或单个订单功能。

### 本项目的核心特点

| 特点 | 说明 |
| --- | --- |
| 本地策略执行 | 价格条件、冷却时间、最大触发次数等逻辑由本地服务控制 |
| 交易信号层 | 规则命中后先生成 `trading_signals`，记录为什么产生交易意图 |
| 自定义风控 | 可以按单笔金额、行情时效、日内次数、日内金额和交易模式做本地风控 |
| 人工确认闸门 | 风控通过后仍需要在 `/triggers` 人工确认，系统不会直接下单 |
| 下单前预览 | 确认前由后端重新计算执行参考价、预估金额、交易规则和风控结果 |
| 全流程审计 | 信号、风控、触发、下单和异常都会写入本地审计日志 |
| 多交易所统一 | OKX 和 Binance 公共能力收敛为统一领域模型，后续可继续扩展 |
| 数据自主 | 关键执行链路沉淀在本地 SQLite，后续可迁移到 PostgreSQL |
| 策略可扩展 | 后续可以接入价格规则之外的信号来源和更复杂的策略模块 |

### 与欧易、币安能力的区别

| 对比项 | 欧易或币安内置能力 | 本项目 |
| --- | --- | --- |
| 定位 | 交易所订单、条件单、机器人或 API 能力 | 本地量化交易执行系统 |
| 触发逻辑 | 由交易所产品或 API 参数托管 | 由本地策略扫描和信号层控制 |
| 风控范围 | 交易所基础校验为主，例如余额、精度、数量、权限 | 可以叠加自定义金额、次数、行情时效和交易模式限制 |
| 确认方式 | 用户在交易所页面或 API 调用时提交 | 策略命中后仍需人工确认 |
| 数据记录 | 订单和成交记录主要在交易所侧 | 本地记录规则、信号、风控、触发、订单和审计日志 |
| 多交易所 | 通常绑定单个交易所生态 | 通过统一模型适配多个交易所 |
| 扩展能力 | 受交易所产品形态限制 | 可按业务需要扩展策略、风控、报表和恢复能力 |

### 与欧易限价委托的关系

欧易限价委托本质上是交易所订单功能，核心是指定交易对、买卖方向、限价价格和数量，然后提交到交易所撮合系统。

本项目当前已经具备生成限价单所需的一部分业务字段，例如：

```text
symbol
side
orderType
limitPrice
baseQuantity
quoteAmount
```

但这些字段目前仍处在本地执行流程中。只有后续接入真实交易、完成签名、余额校验和订单状态同步后，确认下单阶段才会真正调用交易所接口提交限价委托。

更准确的关系是：

```text
欧易限价委托 = 最终可提交的一种订单类型
本项目 = 管理什么时候、为什么、能不能、由谁确认、如何记录这笔订单的执行系统
```

### 交易所内置能力仍然更适合的场景

| 场景 | 原因 |
| --- | --- |
| 简单手动限价单 | 直接使用交易所页面最快 |
| 完全托管条件单 | 条件和订单由交易所维护，本地服务无需常驻 |
| 依赖交易所原生机器人 | 网格、DCA、跟单等成熟产品维护成本低 |
| 高频或强实时撮合 | 订单最终仍需要交易所撮合系统完成 |

### 本项目更适合的场景

| 场景 | 原因 |
| --- | --- |
| 需要多交易所统一视图 | 可以把 OKX、Binance 等能力收敛到统一页面和领域模型 |
| 需要本地风控 | 可按自己的交易纪律限制金额、次数和交易模式 |
| 需要人工确认 | 每次策略命中后由用户确认是否执行 |
| 需要完整审计 | 可以回溯一笔交易从规则到订单的全过程 |
| 需要后续扩展策略 | 可以逐步加入更多信号来源、报表、恢复和重放能力 |

## 当前阶段说明

- 当前执行链路已经具备标准量化执行体的基础结构。
- 当前真实交易仍未开放，订单提交主要走模拟下单。
- 前端展示预估金额、价格、数量只用于辅助理解，最终计算和校验以后端为准。
- 后续接入真实交易前，需要继续完善余额校验、真实下单签名、二次确认保护、订单状态同步和异常恢复。
