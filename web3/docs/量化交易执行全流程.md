# Web3 量化交易执行全流程

本文档用于说明当前项目的整体业务流程和执行逻辑。阅读本文后，应能理解一笔交易从行情、策略、信号、风控、确认、下单、同步、成交、账本、恢复、审计到后续 Polymarket 信号接入的完整流转。

## 1. 系统定位

本项目是本地量化交易执行体，目标是把 OKX、Binance 和后续 Polymarket 的交易相关能力收敛到一套可控流程中。

当前主线：

1. OKX / Binance 行情进入系统。
2. 策略生成交易信号。
3. 风控判断是否允许执行。
4. 用户确认或执行编排确认后进入下单。
5. 模拟交易走本地账本。
6. 真实交易走交易所官方接口。
7. 订单通过私有推送优先同步，REST 作为兜底。
8. 成交后回写成交、账本、持仓、盈亏。
9. 异常进入恢复、审计和后续对账链路。

Polymarket 当前规划为信号前置模块，第一阶段只做市场映射、盘口校验、滞后信号、风控和提醒。自动押注暂不进入当前上线目标。

## 2. 使用前准备

使用前需要确认：

1. 后端和前端服务已启动。
2. SQLite 数据库可用。
3. 默认真实交易开关关闭。
4. 模拟交易可直接使用。
5. 真实交易需要配置 OKX 或 Binance API Key。
6. Binance 建议先使用官方 Spot Test Network 验证。
7. API Key 不得写入前端代码、日志、审计 payload 或 Git 仓库。

真实交易必须同时满足：

1. `ENABLE_REAL_TRADING=true`
2. 风控交易模式允许真实交易
3. 交易所 API Key 配置完整
4. 下单前预检通过
5. 用户确认或执行编排确认通过

## 3. 用户主流程

### 3.1 查看行情和账户状态

用户进入系统后，先查看总览中的行情、账户、持仓和交易状态。

系统会通过 OKX / Binance 官方行情接口和 WebSocket 获取实时价格。实时数据以 WebSocket 为主，REST 只做初始化、低频校准和兜底。

系统会保护行情时序，避免 REST 旧快照覆盖 WebSocket 新价格。

### 3.2 创建策略计划

用户创建交易计划时，需要配置：

1. 交易所
2. 交易对
3. 触发条件
4. 买入或卖出方向
5. 市价或限价
6. 下单金额或数量
7. 下单模式
8. 风控参数
9. 冷却时间
10. 最大触发次数

当前基础能力以价格规则承载策略。后续会升级为统一策略实例模型，策略实例会成为信号、风控、执行、订单、恢复和审计的统一入口。

### 3.3 系统接收行情

系统订阅交易对行情后，会持续维护最新行情快照。

行情用于：

1. 判断策略是否触发。
2. 生成交易信号。
3. 计算下单前预览。
4. 估算持仓收益。
5. 判断行情是否过期。

行情过旧时，风控会拒绝继续执行。

### 3.4 策略生成信号

当市场价格满足策略条件时，系统生成交易信号。

交易信号记录：

1. 信号来源
2. 策略或规则来源
3. 交易所
4. 交易对
5. 市场价格
6. 触发时间
7. 买卖方向
8. 订单类型
9. 下单金额或数量
10. 生成原因

信号是交易意图的第一层业务归档。后续会统一价格规则、外部信号和 Polymarket 信号协议。

### 3.5 风控检查

信号生成后，系统立即进入风控。

当前风控主要检查：

1. 单笔最大金额
2. 每日最大通过次数
3. 每日最大通过金额
4. 行情最大允许延迟
5. 真实交易是否允许
6. 账户余额或可卖数量

风控拒绝后，流程停止，信号保留拒绝原因。

风控通过后，系统进入执行确认或执行编排。

### 3.6 生成执行任务或触发确认

风控通过后，系统生成执行上下文。

当前已有触发确认基础链路。后续会升级为执行任务模型，执行任务会负责：

1. 互斥控制
2. 幂等控制
3. 确认状态
4. 下单状态
5. 失败阶段
6. 恢复关联

同账户、同交易所、同交易对的活跃执行后续需要互斥，避免重复提交。

### 3.7 下单前预览

执行前，后端生成下单预览。

预览会重新读取：

1. 最新行情
2. 交易规则
3. 风控配置
4. 账户余额
5. 持仓可卖数量

预览结果包括：

1. 执行参考价
2. 预计基础币数量
3. 预计计价币金额
4. 预计手续费
5. 成交后余额
6. 成交后持仓
7. 预检明细
8. 真实交易确认令牌

前端只展示预览结果。最终计算和校验以后端为准。

### 3.8 用户确认

用户确认后，后端会再次执行最终校验。

真实交易确认必须满足：

1. 确认令牌有效。
2. 真实交易总开关仍然开启。
3. 风控仍然允许真实交易。
4. 交易规则仍然有效。
5. 账户余额或可卖数量仍然满足要求。

如果最终校验失败，系统不会下单，会写入失败原因和审计记录。

## 4. 模拟交易执行

模拟交易只使用本地账本，不调用交易所私有接口。

### 4.1 模拟买入

流程：

1. 用户确认买入。
2. 后端执行最终校验。
3. 扣减可用 USDT。
4. 增加基础币持仓。
5. 更新平均成本。
6. 写订单记录。
7. 写成交记录。
8. 写操作日志。

### 4.2 模拟卖出

流程：

1. 用户确认卖出。
2. 后端检查可卖数量。
3. 扣减基础币持仓。
4. 增加 USDT 余额。
5. 计算已实现盈亏。
6. 写订单记录。
7. 写成交记录。
8. 写操作日志。

### 4.3 模拟账本口径

账户总权益：

```text
可用余额 + 冻结余额 + 持仓市值
```

浮动盈亏：

```text
当前持仓数量 * 最新市场价 - 当前剩余持仓成本
```

卖出已实现盈亏：

```text
卖出收入 - 本次卖出对应成本 - 手续费
```

## 5. 真实交易执行

真实交易会调用 OKX 或 Binance 官方接口。

### 5.1 OKX 下单

OKX 真实下单以官方 `POST /api/v5/trade/order` 为准。

关键字段以官方文档为准，包括：

1. `instId`
2. `tdMode`
3. `clOrdId`
4. `side`
5. `ordType`
6. `sz`
7. `px`
8. `tgtCcy`

SPOT 市价单金额语义必须按 OKX 官方 `tgtCcy` 规则实现。

### 5.2 Binance 下单

Binance 真实下单以官方 `POST /api/v3/order` 为准。

关键字段以官方文档为准，包括：

1. `symbol`
2. `side`
3. `type`
4. `timeInForce`
5. `quantity`
6. `quoteOrderQty`
7. `price`
8. `newClientOrderId`
9. `timestamp`

Binance 市价单数量语义：

1. `quantity` 表示基础币数量。
2. `quoteOrderQty` 表示计价币金额。

### 5.3 本地订单落库

真实下单成功后，系统必须先写入本地订单记录。

订单记录保存：

1. 内部订单 ID
2. 交易所
3. 交易对
4. 买卖方向
5. 订单类型
6. 数量或金额
7. 客户端幂等 ID
8. 交易所订单号
9. 本地订单状态
10. 响应摘要

订单落库失败会进入恢复任务。

## 6. 订单同步

真实订单提交后，系统通过两条路径同步状态。

### 6.1 私有推送优先

OKX 使用官方私有订单和账户推送。

Binance 使用官方 User Data Stream：

1. `executionReport`
2. `outboundAccountPosition`

私有推送用于尽快同步：

1. 订单状态
2. 成交累计量
3. 余额变化

### 6.2 REST 兜底

REST 用于：

1. 私有推送断线后的兜底。
2. pending 订单定时同步。
3. 成交补全校准。

REST 兜底需要批量限制，避免触发交易所限频。

### 6.3 状态保护

系统需要防止：

1. 旧事件覆盖新状态。
2. 重复推送重复记账。
3. 成交累计量回退。
4. 终态订单被改回非终态。

后续会按 TODO 中的订单状态机继续收口。

## 7. 成交、账本和持仓

订单成交后，系统进入成交补全和账本回写。

流程：

1. 获取订单成交增量。
2. 生成本地成交记录。
3. 更新账户余额。
4. 更新持仓数量。
5. 更新平均成本。
6. 更新已实现盈亏。
7. 写操作日志。
8. 写审计日志。

同一订单的成交补全必须串行处理，避免重复记账。

## 8. 异常恢复和审计

系统出现异常时，会生成恢复任务。

当前恢复阶段包括：

1. 订单提交落库
2. 规则确认收尾
3. 订单同步
4. 私有推送
5. 成交补全
6. 余额刷新

恢复来源包括：

1. 正常链路恢复
2. 自动恢复
3. 人工重试

恢复最终状态包括：

1. 待恢复
2. 恢复中
3. 已恢复
4. 恢复失败
5. 需人工处理

审计日志贯穿：

1. 信号生成
2. 风控通过或拒绝
3. 执行开始
4. 下单提交
5. 订单同步
6. 成交补全
7. 恢复开始和结束
8. 熔断和人工接管

## 9. 对账和熔断

后续上线级能力需要补齐对账闭环。

对账范围：

1. 订单对账
2. 成交对账
3. 余额对账
4. 持仓对账

发现差异后，需要归档差异并进入处理流程：

1. 自动修复
2. 人工确认修复
3. 人工忽略

熔断范围：

1. 全局停机
2. 账户停机
3. 策略停机
4. 连续失败熔断
5. 对账差异熔断

停机后禁止新真实下单，但仍保留订单同步、成交补全、恢复和审计。

## 10. Polymarket 后续接入流程

Polymarket 第一阶段只作为信号前置模块。

### 10.1 V1 市场映射

用户人工配置 Polymarket 市场映射。

映射数据来自官方 Gamma 和 CLOB API。

必须人工确认：

1. 外部行情源
2. Polymarket 市场
3. `conditionId`
4. YES token
5. NO token
6. `resolutionSource`
7. `endDate`
8. 市场是否 `active`
9. 市场是否 `closed`

禁止从 `question` 自动推导阈值、方向、到期条件。

### 10.2 V2 盘口接入

系统订阅 Polymarket Market WebSocket。

重点维护：

1. `book`
2. `price_change`
3. `last_trade_price`
4. `tick_size_change`
5. `best_bid_ask`

滞后判断必须基于订单簿、best bid、best ask、spread 和可成交深度。

### 10.3 V3 信号和风控

OKX / Binance 行情变化后，系统检查 Polymarket 对应盘口。

只有满足以下条件才生成提醒：

1. 市场 active。
2. 市场未 closed。
3. spread 在阈值内。
4. 深度足够。
5. 信号未过期。
6. 最大投入满足风控。
7. 滑点满足风控。

V1 到 V3 只生成提醒信号，不下单。

### 10.4 V4 手动下单预留

后续如果进入 Polymarket 手动下单，必须先完成：

1. geoblock 检查。
2. 官方认证流程。
3. 官方下单字段映射。
4. 用户人工确认。
5. 盘口变化二次校验。
6. 订单、成交、审计归档。

未完成 geoblock、认证和官方字段映射前，禁止开发 Polymarket 交易执行。

## 11. 用户安全边界

真实交易前必须确认：

1. 真实交易总开关已显式开启。
2. 风控允许真实交易。
3. API Key 权限最小化。
4. API Key 没有提现权限。
5. 交易规则已同步。
6. 下单前预览通过。
7. 确认令牌有效。
8. 订单同步可用。
9. 恢复链路可用。
10. 紧急停机可用。

第一版真实上线建议：

1. OKX 只做小额真实交易。
2. Binance 优先使用测试网。
3. Polymarket 只做 V1 到 V3 信号提醒。
4. 自动执行默认关闭。
5. 单策略、单账户、单交易对先验收。

## 12. 最简总流程

```text
用户查看行情和账户状态
-> 创建策略计划
-> 系统接收 OKX / Binance 实时行情
-> 策略生成交易信号
-> 风控检查
-> 生成执行上下文
-> 下单前预览
-> 用户确认
-> 模拟交易或真实交易
-> 订单落库
-> 私有推送优先同步
-> REST 兜底同步
-> 成交补全
-> 更新账本、持仓、盈亏
-> 异常进入恢复
-> 审计和对账归档
```
